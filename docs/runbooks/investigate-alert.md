# Runbook: How to Investigate an Alert Using Logs and Metrics

**Version:** 1.0  
**Last Updated:** 2026-02-21  
**Owner:** SRE Team

---

## Overview

This runbook provides step-by-step instructions for investigating alerts generated by K8sWatch using Grafana dashboards, Prometheus metrics, and Loki logs.

---

## Prerequisites

- Access to Grafana (`http://grafana.observability.svc.cluster.local`)
- Access to Loki for log queries
- Basic understanding of K8sWatch architecture (Agent → Aggregator → AlertManager)

---

## Alert Investigation Workflow

```
1. Receive Alert Notification
         ↓
2. Check Grafana Dashboard
         ↓
3. Analyze Metrics (Prometheus)
         ↓
4. Investigate Logs (Loki)
         ↓
5. Identify Root Cause
         ↓
6. Take Remediation Action
```

---

## Step 1: Receive Alert Notification

### Alert Information to Capture

When you receive an alert, note the following:

| Field | Example | Where to Find |
|-------|---------|---------------|
| Alert ID | `alert-12345` | Notification message |
| Target Name | `postgres-primary` | Notification message |
| Severity | `critical`, `high`, `medium`, `low` | Notification message |
| Blast Radius | `node`, `zone`, `cluster` | Notification message |
| Failure Layer | `L1`, `L2`, `L3`, etc. | Notification message |
| Failure Code | `dns_timeout`, `tcp_refused` | Notification message |
| Correlation ID | `abc123-def456` | Notification message |

### Example Alert Notification

```
[CRITICAL] K8sWatch Alert
Target: postgres-primary
Namespace: production
Failure: L2 TCP Connect
Code: tcp_refused
Blast Radius: zone
Nodes Affected: 5/10 in us-east-1a
Correlation ID: abc123-def456-ghi789
Time: 2026-02-21T10:30:00Z
```

## Step 2: Check Grafana Dashboard

### 2.1 Open Cluster Health Dashboard

1. Navigate to: **Grafana → Dashboards → K8sWatch - Cluster Health Overview**
2. Check the following panels:
   - **Cluster-wide Issues:** Is this a cluster-wide problem?
   - **Blast Radius Distribution:** How many nodes/zones affected?
   - **Check Success Rate:** Overall system health
   - **Top 10 Failing Targets:** Is your target in the list?

### 2.2 Open Target Deep Dive Dashboard

1. Navigate to: **Grafana → Dashboards → K8sWatch - Target Deep Dive**
2. Select:
   - **Namespace:** `production`
   - **Target:** `postgres-primary`
3. Check the following panels:
   - **Current Status:** Red = Unhealthy, Green = Healthy
   - **Success Rate (5m):** Should be > 99%
   - **Failures (1h):** How many failures in the last hour?
   - **Check Latency Percentiles:** Any latency spikes?
   - **Failure Code Distribution:** What's the dominant failure code?
   - **Recent Alerts:** Alert history for this target

### 2.3 Open Node Health Dashboard (if node-related)

1. Navigate to: **Grafana → Dashboards → K8sWatch - Node Health**
2. Select:
   - **Zone:** `us-east-1a` (from alert)
   - **Node:** Specific node if known
3. Check the following panels:
   - **Node Health Status:** Which nodes are unhealthy?
   - **Network Mode Comparison:** Pod vs Host network issues?
   - **File Descriptor Usage:** FD exhaustion?
   - **Conntrack Usage:** Connection tracking pressure?
   - **Clock Skew:** Time synchronization issues?

### 2.4 Open Alerting Metrics Dashboard

1. Navigate to: **Grafana → Dashboards → K8sWatch - Alerting Metrics**
2. Check:
   - **Active Alerts (Firing):** Current alert volume
   - **Notifications by Channel:** Are notifications being sent?
   - **Notification Channel Success Rates:** Any channel failures?

## Step 3: Analyze Metrics (Prometheus)

### 3.1 Check Target Health Metrics

Open **Grafana → Explore** and run:

```promql
# Check success rate for the target
sum(rate(k8swatch_agent_check_total{target="postgres-primary", namespace="production", status="success"}[5m])) 
/ 
sum(rate(k8swatch_agent_check_total{target="postgres-primary", namespace="production"}[5m]))
```

**Expected:** > 0.99 (99%)  
**Alert Threshold:** < 0.95 (95%)

### 3.2 Check Failure Distribution

```promql
# Failures by layer
sum(increase(k8swatch_agent_check_total{target="postgres-primary", status="failure"}[1h])) by (failure_layer)
```

### 3.3 Check Affected Nodes

```promql
# Which nodes are failing?
sum(increase(k8swatch_agent_check_total{target="postgres-primary", status="failure"}[1h])) by (node)
```

### 3.4 Check Latency

```promql
# P95 latency for the target
histogram_quantile(0.95, sum(rate(k8swatch_agent_check_duration_seconds_bucket{target="postgres-primary"}[5m])) by (le))
```

**Expected:** < 1s  
**Alert Threshold:** > 5s

## Step 4: Investigate Logs (Loki)

### 4.1 Open Grafana Explore

1. Navigate to: **Grafana → Explore**
2. Select datasource: **Loki**

### 4.2 Query by Correlation ID (Best Option)

If you have a correlation ID from the alert:

```logql
{namespace="k8swatch"} |= "correlationID=abc123-def456-ghi789"
| json
| sort by timestamp
```

This shows the **full trace** across all components (Agent → Aggregator → AlertManager).

### 4.3 Query by Target Name

```logql
{namespace="k8swatch"} 
|= "level=error" 
|= `target="postgres-primary"`
| json
| line_format "{{.timestamp}} {{.component}} {{.operation}} - {{.message}}"
```

### 4.4 Query by Failure Code

```logql
{namespace="k8swatch"} 
|= "level=error" 
|= "failure_code=tcp_refused"
| json
| line_format "{{.timestamp}} Node={{.node}} {{.error}}"
```

### 4.5 Common Log Patterns

| Issue | LogQL Query |
|-------|-------------|
| DNS failures | `{namespace="k8swatch"} \|= "failure_code=dns_timeout"` |
| TLS errors | `{namespace="k8swatch"} \|= "failure_code=tls_expired"` |
| Auth failures | `{namespace="k8swatch"} \|= "failure_code=auth_failed"` |
| Connection refused | `{namespace="k8swatch"} \|= "failure_code=tcp_refused"` |
| Agent restarts | `{namespace="k8swatch"} \|= "Starting K8sWatch Agent"` |

## Step 5: Identify Root Cause

### Failure Layer Analysis

Use the failure layer from the alert to narrow down the issue:

| Layer | Name | Common Causes | Investigation Focus |
|-------|------|---------------|---------------------|
| L0 | Node Sanity | FD exhaustion, conntrack pressure, clock skew | Node Health dashboard |
| L1 | DNS Resolution | CoreDNS down, DNS misconfiguration | DNS logs, CoreDNS metrics |
| L2 | TCP Connect | Service down, network policy, CNI issue | Target service, network policies |
| L3 | TLS Handshake | Expired cert, wrong hostname, CA mismatch | Certificate expiry, TLS config |
| L4 | Protocol Check | Wrong protocol, service misconfiguration | Application logs |
| L5 | Auth/Authz | Wrong credentials, RBAC issue | Secret config, RBAC rules |
| L6 | Semantic | Application logic error | Application logs |

### Blast Radius Analysis

| Blast Radius | Meaning | Likely Cause |
|--------------|---------|--------------|
| **Node-local** | Single node affected | Node issue, local CNI problem |
| **Zone-level** | Multiple nodes in same zone | Zone outage, zone-specific config |
| **Cluster-wide** | All/most nodes affected | Target outage, DNS issue, central service |

### Decision Tree

```
Is blast radius cluster-wide?
├─ YES → Check target health, DNS, central services
│   ├─ Target down? → Contact target owner
│   ├─ DNS failing? → Check CoreDNS
│   └─ All targets failing? → Check agent/aggregator
│
├─ Zone-level? → Check zone-specific issues
│   ├─ Node egress? → Check NAT gateway
│   ├─ Zone DNS? → Check CoreDNS zone affinity
│   └─ Network policy? → Check zone-specific policies
│
└─ Node-local? → Check single node
    ├─ FD exhaustion? → Restart agent, increase limits
    ├─ CNI issue? → Check CNI pod on node
    └─ Node network? → Check node routing, iptables
```

## Step 6: Take Remediation Action

### Common Remediation Actions

#### 1. Restart Agent on Affected Node

```bash
# Find the agent pod on the node
kubectl get pods -n k8swatch -l app.kubernetes.io/component=agent -o wide | grep <node-name>

# Delete the pod (will be recreated by DaemonSet)
kubectl delete pod -n k8swatch <agent-pod-name>
```

#### 2. Check Target Service

```bash
# Check if target service is running
kubectl get pods -n <namespace> -l app=<target-label>

# Check service endpoints
kubectl get endpoints -n <namespace> <target-service>

# Test connectivity from a pod
kubectl run -it --rm debug --image=curlimages/curl --restart=Never -- \
  curl -v http://<target-service>.<namespace>.svc.cluster.local:<port>
```

#### 3. Check DNS

```bash
# Check CoreDNS pods
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Test DNS resolution
kubectl run -it --rm dns-test --image=busybox:1.36 --restart=Never -- \
  nslookup <target-service>.<namespace>.svc.cluster.local
```

#### 4. Check Network Policies

```bash
# List network policies in namespace
kubectl get networkpolicies -n <namespace>

# Check if policy is blocking traffic
kubectl describe networkpolicy <policy-name> -n <namespace>
```

#### 5. Check Certificates (TLS Issues)

```bash
# Check certificate expiry
kubectl get secret <tls-secret> -n <namespace> -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -noout -dates
```

## Escalation

### When to Escalate

| Condition | Escalate To |
|-----------|-------------|
| Cluster-wide outage > 5 min | On-call SRE Lead |
| Target owner needed | Application Team |
| Network/CNI issue | Network Team |
| Certificate renewal needed | Security Team |

### Escalation Template

```
Subject: [ESCALATION] K8sWatch Alert - <Target Name> - <Severity>

Alert Details:
- Target: <target-name>
- Namespace: <namespace>
- Severity: <severity>
- Blast Radius: <blast-radius>
- Failure Layer: <layer>
- Failure Code: <code>
- Correlation ID: <correlation-id>
- Start Time: <time>
- Affected Nodes: <count>

Investigation Summary:
- Dashboards checked: [ ] Cluster Health [ ] Target Deep Dive [ ] Node Health
- Metrics analyzed: <summary>
- Logs reviewed: <summary>
- Root cause hypothesis: <hypothesis>

Actions Taken:
1. <action>
2. <action>

Request:
<what you need from the escalatee>
```

## Post-Incident

### After Resolution

1. **Verify alert resolved:**
   - Check Grafana dashboard shows green
   - Confirm alert is cleared in AlertManager

2. **Document findings:**
   - Update incident ticket
   - Add to runbook if new issue type

3. **Create follow-up tasks:**
   - Fix root cause (if not done)
   - Add monitoring for similar issues
   - Update alerting rules if needed

## Appendix: Failure Codes Reference

| Code | Layer | Meaning | Common Cause |
|------|-------|---------|--------------|
| `dns_timeout` | L1 | DNS query timeout | CoreDNS down, network issue |
| `dns_nxdomain` | L1 | Domain not found | Wrong hostname, DNS misconfiguration |
| `tcp_timeout` | L2 | TCP connect timeout | Service down, firewall |
| `tcp_refused` | L2 | TCP RST received | Service not listening |
| `tls_expired` | L3 | Certificate expired | Certificate renewal needed |
| `tls_hostname` | L3 | Certificate hostname mismatch | Wrong certificate |
| `auth_failed` | L5 | Authentication failed | Wrong credentials |
| `protocol_error` | L4 | Unexpected protocol response | Service misconfiguration |
